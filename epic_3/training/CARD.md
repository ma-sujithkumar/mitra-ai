# Technical Architecture: Local Training Worker

## Overview
The `training` submodule implements local execution of single model training tasks. It isolates the model wrapper instantiation, dataset loading, and parameter feeding from Ray coordination and HTTP lifecycle.

## Core Component Walkthrough
1. **`data_loader.py`**: Handles low-level reading of `.csv` or `.npz` files into NumPy arrays (`X_train`, `y_train`, `X_test`, `y_test`). It checks for column alignment, target presence, and shape compatibility.
2. **`library_adapter.py`**: Instantiates a wrapper class from `model_library.ml_kit.MODEL_REGISTRY`. It passes configuration hyperparameters, calls `.fit(X, y)`, and computes predictions.
3. **`metrics.py`**: Evaluates model predictions on train/test sets to generate F1, F1-macro, Accuracy, MSE, MAE, R², etc.
4. **`artifact_writer.py`**: Handles directory creation and dumps metrics and weights.
5. **`trainer.py`**: Coordinates the data loader, adaptor, evaluator, and artifact writer.

## Interfacing Guide
- **Upstream Integration:** Receives job description configurations created by the `training_orchestrator` or generated by Optuna trials.
- **Downstream Integration:** Stores binary weights and metrics inside session-scoped model directories, which are then consumed by overfitting analysis or the Judge agent.

## Suggested Cleanup/Refactoring
- **Image Dataset Support:** Current `data_loader.py` handles numeric tabular data only. Support for loading images from directory walks is needed for CNN pipelines.
- **Unified Serialization:** Standardize model checkpointing format (e.g., ONNX or safe-tensors instead of pickling Python classes) to avoid serialization bugs.
