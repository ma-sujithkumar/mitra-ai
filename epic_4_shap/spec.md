SPECIFICATION DOCUMENT
Epic 4 – SHAP Explainability Module
Version: 1.0
________________________________________
1. Objective
Develop a standalone SHAP Explainability Module that consumes outputs from:
•	Epic 2 (Feature Engineering)
•	Epic 3 (Model Training)
and generates:
•	SHAP explainability visualizations
•	Feature importance metrics
•	Feature-to-SHAP mapping datasets
•	Structured CSV outputs
•	Metadata reports
•	Execution logs
The module shall explain trained machine learning models without modifying, retraining, or optimizing them.
________________________________________
2. Scope
The module is responsible only for model explainability.
The module SHALL NOT:
•	perform feature engineering
•	train models
•	tune hyperparameters
•	modify model artifacts
•	perform causal inference
•	make business recommendations
•	alter datasets
The module SHALL only explain model behavior using SHAP.
________________________________________
3. Epic Integration Contract
Epic 2 Output
Provides:
•	Feature-engineered dataset
Artifact:
engineered_dataset.csv
The engineered dataset shall be considered the authoritative source for:
•	feature names
•	feature ordering
•	feature values
________________________________________
Epic 3 Output
Provides:
•	Trained machine learning model
Artifact:
model.pkl
________________________________________
Epic 4 Responsibility
Consume Epic 2 and Epic 3 artifacts and generate SHAP explainability outputs.
________________________________________
4. Input Contract
Epic 4 SHALL consume the integration payload provided by upstream epics.
The JSON contract SHALL NOT be modified by Epic 4.

The SHAP module shall consume a JSON payload provided by upstream epics.

Required Fields:

| Field Name | Type | Description |
|------------|------|-------------|
| session_id | string | Unique execution identifier |
| model_name | string | Model name supplied by Epic 3 |
| pickle_file_path | string | Path to trained model artifact |
| engineered_dataset_path | string | Path to engineered dataset produced by Epic 2 |

Example:

```json
{
"session_id": "session_001",
"model_name": "xgboost",
"pickle_file_path": "/models/model.pkl",
"engineered_dataset_path": "/data/engineered_dataset.csv"
}
```

5. Output Configuration
The module shall store all generated artifacts in a configured output directory.
The output root directory shall be maintained as an application configuration parameter and shall not be supplied through the integration payload.
The exact default output path remains configurable and may be finalized during deployment and integration.
The system shall:
•	create output directories if missing
•	create session-specific folders
•	organize outputs using session_id
Example:
<configured_output_root>/
    └── <session_id>/

6. Supported Models
Phase 1 support:
•	XGBoost
•	Random Forest
•	LightGBM
•	CatBoost
•	Logistic Regression
Unsupported models shall generate descriptive error messages.
6.1 Supported Prediction Types
Phase 1 support includes:

- Binary Classification
- Multiclass Classification
- Regression

The SHAP module shall automatically determine the prediction type from the loaded model where possible.

If automatic detection is not possible, prediction type shall be inferred from available model metadata.

The detected prediction type shall be recorded in metadata.json.

Example values:

- binary_classification
- multiclass_classification
- regression
________________________________________
7. Model Loading and Validation
The system shall:
1.	Validate model file existence
2.	Load model artifact
3.	Detect actual model type
4.	Validate model compatibility
5.	Validate dataset compatibility
Execution shall stop on validation failure.
________________________________________
8. Model Type Validation
The detected model type shall always be used for SHAP explainer selection.
The supplied model_name shall be used for logging, metadata, and integration traceability only.
The supplied model_name shall be treated as integration metadata.
Validation Rules:
Rule 1
If supplied model_name matches detected model type:
•	continue execution
Rule 2
If supplied model_name differs from detected model type:
•	log warning
•	record discrepancy in metadata
•	continue execution
The detected model type SHALL take precedence.
Rule 3
If model type cannot be determined:
•	log error
•	terminate execution
Rule 4
If detected model type is unsupported:
•	log error
•	terminate execution
________________________________________
9. Dataset Validation
The system shall validate:
•	dataset exists
•	dataset is readable
•	dataset is not empty
•	feature count > 0
Validation failures shall terminate execution.
________________________________________
10. Dataset Feature Schema
The engineered dataset supplied by Epic 2 shall be considered the authoritative source of:
•	feature names
•	feature ordering
•	feature values
The SHAP module shall derive all feature metadata from the engineered dataset.
Model metadata may be used for validation only.
The SHAP module shall not depend on feature names being present inside the model artifact.
________________________________________
11. Target Column Handling
The engineered dataset may contain:
•	feature columns
•	optional target column
Before SHAP processing:
1.	Target column shall be identified if present.
2.	Target column shall be excluded from SHAP processing.
3.	Only feature columns shall be supplied to the SHAP explainer.
4.	Feature ordering shall remain consistent with model expectations.
Target column identification strategy shall be configurable.
________________________________________
12. Dataset and Model Compatibility Validation
Before SHAP execution, the system shall validate compatibility between:
•	engineered dataset
•	loaded model
Validation checks:
•	feature columns exist
•	feature count is valid
•	schema is usable by the loaded model
Where model metadata is available:
•	feature names shall be compared
•	feature counts shall be compared
Compatibility failures shall:
•	log errors
•	terminate execution
________________________________________
13. Dynamic Feature Handling
The SHAP module shall never hardcode feature names.
All feature names shall be dynamically extracted from:
•	engineered dataset schema
•	model metadata (if available)
The system shall support arbitrary feature names generated by Epic 2.
All plots, CSV files, metadata, and logs shall use dynamically discovered feature names.
________________________________________
14. SHAP Explainer Selection
Automatic SHAP explainer selection shall be performed.
Model Type	SHAP Explainer
XGBoost	TreeExplainer
Random Forest	TreeExplainer
LightGBM	TreeExplainer
CatBoost	TreeExplainer
Logistic Regression	LinearExplainer
The mapping shall be internal and automatic.
________________________________________
15. SHAP Processing Workflow
The system shall perform:
1.	Load engineered dataset
2.	Load trained model
3.	Detect model type
4.	Validate model name
5.	Validate dataset
6.	Validate schema compatibility
7.	Select SHAP explainer
8.	Generate SHAP values
9.	Calculate feature importance metrics
10.	Generate visualizations
11.	Export CSV outputs
12.	Export metadata
13.	Export logs

17.2 Feature-SHAP Mapping Export

Filename:

feature_shap_mapping.csv

Purpose:

Provide a machine-readable mapping between:
-record
-feature
-feature value
-SHAP contribution

This file shall be considered the primary machine-readable explainability artifact.

The file shall contain both:

original feature value
corresponding SHAP contribution

for every record-feature combination processed by the SHAP module.

Output Schema

For Binary Classification and Regression

Required Columns:

record_id,feature_name,feature_value,shap_value

Illustrative Example Only:

1,feature_A,35,0.120
1,feature_B,1200,-0.081

2,feature_A,42,0.305
2,feature_B,950,-0.154

For Multiclass Classification

Required Columns:

record_id,class_name,feature_name,feature_value,shap_value

Illustrative Example Only:

1,class_0,feature_A,35,0.120
1,class_0,feature_B,1200,-0.081

1,class_1,feature_A,35,-0.045
1,class_1,feature_B,1200,0.091
Additional Rules
Actual feature names and feature values shall be dynamically extracted from the engineered dataset.
Feature names shall never be hardcoded.
One record-feature combination shall generate one row for Binary Classification and Regression.
One record-feature-class combination shall generate one row for Multiclass Classification.
SHAP values shall be exported using the full available numerical precision unless otherwise configured.

}

23. Explainability Disclaimer
The module explains model behavior only.
SHAP outputs describe:
"How the trained model uses input features to generate predictions."
SHAP outputs shall not be interpreted as:
•	causal effects
•	intervention effects
•	policy recommendations
•	business actions
The module performs predictive explainability only and does not perform causal inference.

25. Assumptions
A1. Epic 2 provides a valid feature-engineered dataset.
A2. The engineered dataset schema matches the schema used during model training.
A3. Epic 3 provides a valid serialized model artifact.
A4. The model artifact can be loaded using supported serialization libraries.
A5. The engineered dataset is available before SHAP execution begins.
A6. Supported model types are limited to those listed in Section 6.
A7. The execution environment has sufficient memory and storage to load the model and dataset.
A8. SHAP library version remains compatible with supported model types.
A9. The engineered dataset represents the feature space expected by the trained model.
A10. The engineered dataset contains all feature columns required by the trained model.
A11. Model artifact is serialized using a supported loading mechanism
(joblib or pickle).
