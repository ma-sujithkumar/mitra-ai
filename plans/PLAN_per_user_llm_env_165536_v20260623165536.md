# Plan: Per-user LLM settings persistence + real user-identity enforcement

Date: 2026-06-23
Decisions locked in with user before this plan was written:
- User identity derived from real auth (token), not a client-supplied field.
- Directory: `.mitra/<user_id>/.env` (dot-prefixed, matches existing `.mitra/` convention).
- Smoke test response returns the real API key value (frontend renders it in a `type="password"` input) — no masking/last-4 logic needed.
- Persist the full LLM settings bundle (provider, model, gatewayUrl, apiKey) on smoke test, not just the key.

## Why this is two changes, not one

The literal ask ("store api_key in `.mitra/<user_id>/.env`") is blocked on a prerequisite: **no backend route today knows who is calling it.** `auth.py` validates a password and returns a `user_id` string, but issues no token — every other router (`llm.py`, `feature_engineering.py`, `metadata.py`, `training.py`, ...) accepts no identity at all. So this plan has two parts:

1. Minimal real auth: issue an opaque session token on login, validate it on the routes that touch LLM settings.
2. Per-user `.env` resolution/persistence wired through the existing `LlmSettingsResolver` chokepoint.

## Verified facts (from codebase scan, not assumptions)

- `LlmSettingsResolver` (`backend/agents/metadata_gen_agent.py:444-566`) is the single chokepoint nearly everything funnels through (`judge_agent.py`, `run_pipeline.py`, `routers/feature_engineering.py`, `routers/metadata.py`, `agents/feature_engineering/cli.py`). One exception: `backend/routers/health.py:22` calls `dotenv_values()` independently — must be fixed too.
- No `load_dotenv()`/caching anywhere; `.env` is read fresh via `dotenv_values()` on every resolver call — switching the path per-request is cheap, no cache invalidation needed.
- `.mitra/` and `.env` are already in `.gitignore` — new per-user `.env` files need no extra ignore rule.
- `backend/session.py:78` (`list_recent_uploads`) iterates `workspace_root.iterdir()` but skips any entry lacking `session.json` + `data/data.csv` — confirms a `.mitra/<user_id>/` directory (which won't have those files) is silently skipped, not misread as a session. Safe to place user dirs directly under `.mitra/`.
- `passlib`/`bcrypt` are already installed and used for password hashing in `backend/auth/service.py`. No JWT library is installed. To avoid adding a new dependency (CLAUDE.md: ask before installing anything new), default to an **opaque server-side session token** stored in a new `sessions` table in the existing auth DB (SQLite/Postgres via `backend/auth/db.py`), not a JWT. Flag this as a default the user can override if they specifically want JWT.
- Frontend has one central request wrapper, `requestJson()` in `frontend/src/api/client.js:13-22` — the correct single place to attach an `Authorization` header once a token exists, rather than touching every call site.
- `frontend/src/App.jsx` already persists `mitra.authUser` to localStorage (lines ~54-66) — extend that object with the token rather than introducing a second storage key.

## Part 1 — Minimal real auth (prerequisite)

1. `backend/auth/models.py`: add a `Session` table (`token` PK or unique-indexed, `user_id`, `created_at`, `expires_at`).
2. `backend/auth/service.py`: on successful `login()`/`signup()`, generate `secrets.token_urlsafe(32)`, insert a `Session` row, return it alongside the existing `user_id`/`workspace_path` payload.
3. `backend/auth/db.py` / new `backend/auth/dependencies.py`: add a FastAPI dependency `get_current_user_id(request)` that reads the token from a request header (e.g. `Authorization: Bearer <token>`), looks up the `Session` row, checks expiry, returns `user_id` or raises 401.
4. Apply this dependency to the routes that need user identity for LLM settings: `backend/routers/llm.py` (smoke-test + new settings GET, see Part 2), and the job-start endpoints that kick off long-running pipelines (`upload`, `feature_engineering` start, `training` start) so `user_id` can be captured once and stored in that run's session/state for later background use.
5. Frontend: extend the `mitra.authUser` object stored in `App.jsx` with the returned token; update `requestJson()` in `client.js` to attach `Authorization: Bearer <token>` automatically when a token is present.

This does not productionize auth fully (no token rotation/refresh, no rate limiting) — it's scoped to "enough identity to know whose `.env` to touch," consistent with what was asked. Flag separately if full hardening is wanted later.

## Part 2 — Per-user `.env` resolution and persistence

6. `backend/agents/metadata_gen_agent.py`: change `LlmSettingsResolver.__init__` to accept an optional `user_id`. Resolve `env_path` as:
   - `repo_root / ".mitra" / user_id / ".env"` if `user_id` is given and that file exists,
   - else `repo_root / ".env"` (existing root fallback, unauthenticated/legacy behavior unchanged).
   No other change to the precedence logic (per-run arg > resolved `.env` > config.ini) — minimizes blast radius.
7. `backend/routers/llm.py`:
   - `/api/llm/smoke-test`: add the auth dependency, pass `user_id` into the resolver. On a **successful** smoke test only, write the full resolved bundle (`provider`, `model`, `gatewayUrl`, `apiKey`) to `.mitra/<user_id>/.env`, creating the directory (`mkdir -p`) and the file with `0600` permissions if it doesn't exist.
   - New `GET /api/llm/settings` endpoint (doesn't exist today): auth-dependency-protected, returns the resolved bundle for the current user (this is what the Settings page calls on mount to pre-populate the form, including the real `apiKey` per the locked-in decision).
8. `backend/routers/health.py:22`: stop calling `dotenv_values()` directly; route through `LlmSettingsResolver` (with `user_id=None`, i.e. root `.env` only — health checks aren't user-scoped) so there's exactly one place that ever reads an `.env` file.
9. Background/long-running jobs (training run, judge loop, feature-engineering pipeline): capture `user_id` at job-start time (from the auth dependency on the trigger endpoint) and store it in the run/session state object that's already threaded through (`SessionInfo`/training run state). Every later call to `LlmSettingsResolver` deep in `judge_agent.py` / `run_pipeline.py` reads `user_id` from that state, not from a live request — this avoids needing request context inside background tasks.
10. Frontend: `Settings.jsx` calls the new `GET /api/llm/settings` on mount, populates `llmSettings` (already exists in `App.jsx`) including the real `apiKey`; the existing `ByomFields.jsx` apiKey input already needs to switch to `type="password"` (currently likely plain text — verify and fix as part of this step).

## Explicit non-goals / things intentionally left alone

- No auto-migration of an existing root `.env`'s content into any specific user's directory — multiple users could exist, there's no single correct owner. Root `.env` remains the fallback for any user who hasn't run a smoke test yet.
- No token refresh/rotation/logout-invalidation beyond a basic expiry — out of scope unless asked.
- `agents/metadata_gen_agent.py:620` / `domain_reasoning_agent.py:150`'s hardcoded `"mitra_epic1"`-style `user_id` literals are an unrelated ADK session-bookkeeping concept, not the auth user — left untouched, would be a separate task if it needs fixing.

## Test plan

- Spawn an Agent to run `backend/tests/` after changes (per CLAUDE.md rule 26), specifically any existing `test_auth*.py`/`test_llm*.py`, plus new tests for: token issuance/validation, resolver path-selection (`user_id` given + file exists / given + file missing / not given), and smoke-test-writes-on-success-only behavior.
- Manual frontend check: log in as two different users, set different keys via Settings, confirm each user's Settings page reloads with their own key (not the other's, not root `.env`'s).
