  ---
  Risks (these block implementation or produce wrong results)

  R1 ‚ÄĒ The Input Format contradicts the Requirements. This is the biggest one.
  The input is {train_metrics, test_metrics} ‚ÄĒ already-computed metrics. But:
  - Requirement 1 (gap = train ‚ąí test) works fine from metrics alone. ‚úÖ
  - Requirement 2 (K-fold CV) is impossible from metrics alone. K-fold needs the
  raw dataset (X, y) and the ability to re-train the model on each fold. You
  cannot do K-fold from two summary dicts.
  - Requirement 3 ("interface with a training class") confirms K-fold must
  actually run training.

  So the spec asks for an input that makes half of it unimplementable. We must
  decide: the tool receives a model handle + data (so it can run training/CV),
  and metrics become optional precomputed inputs ‚ÄĒ not the only input.

  R2 ‚ÄĒ Reinvention risk. metrics/evaluators.py::compute_metrics and the
  MetricsResult dataclass already compute accuracy/f1/precision/recall
  (classification) and mse/rmse/mae/r2 (regression). ml_kit.py::MLKit already
  does .train() / .test() over
  DataBundle(CommonData(X_train,y_train,X_test,y_test)). The spec should
  explicitly reuse these (CLAUDE.md #1, #28) instead of a placeholder + custom
  metric code.

  R3 ‚ÄĒ Metric direction is unhandled. "gap = train ‚ąí test" only makes sense for
  higher-is-better metrics (accuracy, f1, r2). For mse/rmse/mae
  (lower-is-better), overfitting means test > train, so the sign flips. A naive
  subtraction will report regression overfitting backwards. Needs a
  direction-aware map.

  R4 ‚ÄĒ No reproducibility spec. No random seed for the K-fold split ‚Üí
  non-reproducible verdicts in an AutoML pipeline.

  ---
  Ambiguities (must be pinned down before coding)

  A1 ‚ÄĒ "Overfitted or not" has no decision rule. The GOAL says decide if the
  model overfit, but Requirements and Output have no threshold and no verdict
  field. There's no is_overfitted boolean and no configurable threshold.

  A2 ‚ÄĒ Regression "accuracy" undefined. Req 1 literally says "training accuracy
  ‚ąí test accuracy", but regression has no accuracy. Which metric defines the gap
  per task type? (classification ‚Üí accuracy or f1? regression ‚Üí r2? rmse?)

  A3 ‚ÄĒ train_metrics / test_metrics schema is {} ‚ÄĒ keys unspecified. Should they
  match MetricsResult field names? If multiple metrics are present, is there
  one gap or a gap per metric?

  A4 ‚ÄĒ K-fold parameters all unspecified: value of k, StratifiedKFold vs KFold,
  which metric is scored per fold, scored on train-only or full data, and the
  output schema of k_fold_cross_validation_results (per-fold list? mean? std?).

  A5 ‚ÄĒ Where does data come from for K-fold? Input json (a small metrics file)
  has no X/y and no data path. CLI -i <input_json> can't carry a dataset.
  Unresolved against R1.

  A6 ‚ÄĒ Training-class interface contract undefined. "keep a placeholder" ‚ÄĒ but

     Class OverfittingAnalyzer:
     - __init__(self, input_json_path: str, output_dir: str, verbose: bool) -> 
     None
     Reads config.ini + config.yaml (own loader, configparser+yaml). Class vars
     hold config
     sub-dicts shared across methods (CLAUDE.md #9). Parses input JSON;
     validates required keys
     (model_type, model_name, dataset_path); calls validate_model_name; checks
     model_name
     vs model_type consistency against MODEL_REGISTRY groupings.
     - load_dataset(self, dataset_path: str) -> CommonData
     np.load(.npz), require arrays X_train,y_train,X_test,y_test; error clearly
     if absent.
     - compute_holdout_metrics(self, common: CommonData) -> 
     tuple[MetricsResult|None, MetricsResult|None]
     If train_metrics/test_metrics present in input -> wrap into
     MetricsResult-shaped dicts and
     use as-is. Else: one MLKit train run; test() -> test preds;
     kit.model.predict(common.X_train)
     -> train preds; compute_metrics for each. Returns (train_result,
     test_result).
     - run_kfold(self, common: CommonData) -> dict
     Assemble CV X,y per cv_data_source. StratifiedKFold (classification) or
     KFold (regression)
     with k/shuffle/random_state. Per fold: build CommonData (train idx ->
     train, val idx -> test),
     DataBundle, fresh MLKit(model_name), train(), test(), compute_metrics, pull
     the
     configured scoring_metric. Return {k, scoring, per_fold_scores, mean, std, 
     train_vs_cv_gap},
     where train_vs_cv_gap is the direction-aware gap of holdout train scoring
     metric vs cv mean.
     Validate k <= min class count (classification) / k <= n_samples; error
     clearly otherwise.
     On retrain failure, return None and set cv_skipped_reason.
     - compute_gaps(self, train_result, test_result) -> tuple[dict, float|None]
     Iterate non-None metrics for the task; gap via
     DIRECTION_GAP_FUNCS[metric_direction_map[m]].
     Regression also computes rel_rmse_gap = (test_rmse - 
     train_rmse)/max(train_rmse, epsilon).
     - decide_verdict(self, gaps: dict, kfold: dict|None) -> bool
     primary_gap = gaps[primary_metric] when holdout available -> is_overfitted 
     = primary_gap > gap_threshold.
     If holdout metrics absent, fall back to kfold["train_vs_cv_gap"] > 
     gap_threshold.
     - write_output(self, payload: dict) -> str
     mkdir -p output_dir (CLAUDE.md #17); write
     <output_dir>/overfitting_analysis.json matching SPEC Section¬†6.
     - run(self) -> dict orchestrates the above and returns the payload.
