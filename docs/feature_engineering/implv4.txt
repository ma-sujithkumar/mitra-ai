● Feature selection is the most agentic step in the pipeline because it
  delegates the actual method choice to the Judge sub-agent rather than choosing
   locally. End-to-end flow:

  1. Evidence assembly (pipeline/tools/selector.py:32-77)

  The selector takes the current dataframe (which includes FeatureCreator-added
  columns — the cached profile clusters are stale at this point, so they're
  recomputed) and produces a FeatureSelectorEvidence packet:

  - compute_correlation_clusters(X, 
  cut_threshold=cfg.feature_selection.cluster_cut_threshold) at parallel.py:104
  — average-linkage hierarchical clustering on 1 - |corr|, cut at 0.30 (config).
   Returns {cluster_id: [columns]}.
  - compute_linear_baseline(X, y, task, k=linear_baseline_k) at parallel.py:140
  — fits LogisticRegression/LinearRegression on the top-k MI features, returns
  CV-AUC (binary classification) / CV-accuracy (multi-class) / CV-R²
  (regression). Cheap proxy for "do linear methods work on this data?"
  - Per cluster: mean_mi, max_mi against the target, and intra_cluster_corr
  (mean off-diagonal |corr| within the cluster).

  2. Plan request to Judge (selector.py:79-86)

  Selector hands the packet to self.judge.plan(evidence, cfg). The Judge
  sub-agent has its own isolated ADK Agent + InMemoryRunner
  (judge_agent.py:217-265) so its reasoning never leaks into the orchestrator
  context.

  Judge's prompt is JUDGE_PLAN_INSTRUCTION at judge_agent.py:38. It lists the
  six allowed actions with mechanical-only descriptions (per spec §5 — no "use
  mrmr when X" guidance, only what each action mechanically does), and asks for
  a JSON object shaped:

  {"plan": [
    {"cluster_id": 0, "action": "mrmr_then_pca",
     "rationale": "...≥60 chars citing EVIDENCE fields...",
     "evidence_cited": ["clusters.mean_mi", "clusters.intra_cluster_corr"],
     "alternatives_considered": ["mrmr", "drop"]}
  ]}

  The response is parsed through validate_response(SelectionPlanResponse, ...)
  (responses.py:175) — same four content checks as every other tool. One
  revision retry on failure; second failure → plan() returns None.

  3. Per-cluster execution (selector.py:97-128)

  The plan is a list of (cluster_id, action) pairs. Selector walks the clusters
  and runs the matching action on the cluster's columns only:

  Action: mrmr
  Implementation: mrmr_classif / mrmr_regression from the mrmr package
  Output: cluster_k original columns
  ────────────────────────────────────────
  Action: pca
  Implementation: PCA(n_components=cluster_k, random_state=42)
  Output: new pca_c{hash}_i columns, originals discarded; lineage logged to
    state.created_columns + state.transformers
  ────────────────────────────────────────
  Action: mrmr_then_pca
  Implementation: half budget to mRMR over the cluster, PCA over the residual
    columns for the other half
  Output: mRMR survivors + PCA components
  ────────────────────────────────────────
  Action: drop
  Implementation: no-op
  Output: empty (cluster discarded)
  ────────────────────────────────────────
  Action: lasso
  Implementation: L1 logistic (classification) or Lasso (regression), top-k by
    |coef|
  Output: original columns
  ────────────────────────────────────────
  Action: rf_importance
  Implementation: RandomForest, top-k by feature_importances_
  Output: original columns

  Per-cluster budget = top_k_features * cluster_size / total_features so big
  clusters get more budget. Implementation at selector.py:104.

  Anything that fails mid-cluster (e.g. mrmr package missing, PCA infeasible)
  falls back to per-cluster top-MI — selector.py:127.

  4. Aggregation (selector.py:131-142)

  - All survivors are concatenated, de-duplicated (PCA names are deterministic
  but mRMR could return overlap with prior clusters).
  - Capped at top_k_features.
  - state.selection_method records the plan, e.g.
  "plan:[c0:mrmr_then_pca,c1:pca,c2:drop,c3:lasso]".
  - state.selected_columns is the final list.

  5. Fallbacks (selector.py:88-95, selector.py:136)

  - Judge unavailable or both attempts fail content checks →
  "fallback:mrmr_all": run mRMR over the entire feature set with top_k_features
  from config.
  - mRMR package not installed → _top_mi: pure MI ranking.
  - mRMR returns nothing → first-k columns as the absolute-last-resort with
  "fallback:first_k".

  6. Downstream wiring (tools/adk_tools.py:39-44)

  The orchestrator constructs FeatureSelector(model_call, judge=judge_agent)
  once at startup, so the judge is the same instance used by
  FeatureCreator.rank() for proposal ranking — single isolated sub-agent, two
  entry points (rank and plan).

  So the model only sees one prompt during selection (the judge's plan request),
   and code does everything else. The selector itself never calls the
  orchestrator's model.
